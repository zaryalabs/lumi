# База знаний

Status: draft

## Контекст

База знаний в Lumi - это простая Obsidian-like среда для заметок, связей и
выжимок из чтения. Она не должна конкурировать с Obsidian как полноформатный
редактор с сотнями плагинов. Ее задача - замкнуть цикл чтения:

```text
Материал -> хайлайт/заметка -> осмысление -> связанная Markdown-запись
  -> граф идей -> поиск/повторение/ИИ
```

Базовая форма записи - Markdown документ. Поддерживаются wikilinks, tags,
backlinks, вложения, обратные ссылки на материалы и anchors. Главная
заимствованная из Obsidian идея - graph: пользователь должен видеть связи
между заметками, материалами, понятиями и generated artifacts.

## Пользовательские сценарии

- Пользователь создает заметку в базе знаний как обычный Markdown документ.
- Пользователь выделяет фрагмент в книге и превращает его в KB note с quote,
  source link и backlinks.
- Пользователь пишет `[[понятие]]`, Lumi создает unresolved link или предлагает
  существующую запись.
- Пользователь открывает backlink panel и видит, какие материалы, заметки,
  summaries и карточки ссылаются на текущую идею.
- Пользователь смотрит graph по всей базе или по выбранной папке/material.
- Пользователь ищет по базе знаний вместе с книгами, хайлайтами, summary и
  learning artifacts.
- Пользователь экспортирует базу в Markdown folder или синхронизирует ее с
  Obsidian vault.
- ИИ создает черновик summary, карточки, понятия или связи, а пользователь
  принимает их в базу знаний.

## Функциональные требования

### Заметки и папки

- KB состоит из Markdown notes и attachments.
- Notes могут лежать в папках.
- Папки являются организацией, но не единственным способом связей: tags и
  wikilinks важнее для graph.
- Одна note имеет stable `kb_note_id`, title, body, front matter, folder,
  tags, created_at, updated_at и sync revision.
- File name/path является projection/export detail, not primary id.
- Note может быть:
  - user-authored;
  - generated draft;
  - accepted AI artifact;
  - imported from Obsidian/filesystem;
  - exported snapshot of reader annotations.

### Markdown dialect

KB использует `lumi-markdown`, описанный в
[`formats/markdown.md`](formats/markdown.md):

- CommonMark + GFM baseline;
- YAML/TOML front matter;
- wikilinks;
- tags;
- callouts;
- local resources и embeds.

KB editor не должен исполнять raw HTML/JS. Unsupported content becomes
placeholder или escaped text, как в Markdown importer.

### Links and graph

Поддерживаемые связи:

- `[[note]]` - note link.
- `[[note#heading]]` - heading link.
- `[[note|alias]]` - alias.
- `[[material:title]]` или UI-created link to material.
- Link to reader anchor: source material, document revision, quote и anchor.
- Tags: `#tag` и front matter tags.
- Generated concept links from AI/entity extraction.
- Attachment embeds: images, PDFs, audio, datasets.

Backlinks и graph indexes - derived data. Их можно перестроить из notes,
materials, annotations и accepted AI artifacts.

### Sources and provenance

KB item должен сохранять происхождение:

- created manually;
- created from highlight/note;
- generated by AI task;
- imported from Obsidian;
- transformed from learning artifact.

Для reader-derived notes сохраняем:

- `material_id`;
- `document_revision_id`;
- `anchor`;
- quote;
- title/author/source metadata;
- created_at;
- optional user note/comment.

Provenance важен для экспорта, поиска, социальных сценариев и copyright-safe
sharing.

### Graph

Graph должен включать:

- KB notes;
- materials;
- highlights и annotations;
- concepts/entities;
- tags;
- AI artifacts accepted into KB;
- learning items, если они стали постоянными knowledge artifacts.

Edge types:

- `links_to`;
- `backlink`;
- `mentions`;
- `tagged_with`;
- `derived_from`;
- `same_source`;
- `supports`;
- `contradicts` as future/manual relation;
- `related_by_ai` as low-confidence generated relation.

Graph UI должен уметь показывать:

- local graph around current note/material;
- whole vault graph with filters;
- isolated unresolved links;
- source-backed paths from note to original reading anchor.

### Generated artifacts

ИИ может создавать:

- summaries;
- extracted concepts/entities;
- question sets;
- flashcards;
- link suggestions;
- note drafts;
- chapter outlines.

По умолчанию generated artifact не равен KB note. Он становится KB note только
после user action или explicit auto-accept policy.

### Редактирование

Минимальный editor:

- plain Markdown editing;
- preview through same Markdown renderer/reader-compatible view;
- wikilink autocomplete;
- backlink panel;
- source link picker;
- tag management;
- attachments insertion.

Не требуется на первом проходе:

- полноценная Obsidian command palette;
- plugin ecosystem inside editor;
- realtime collaborative editing of KB notes.

## Нефункциональные требования

- **Portability.** Notes export as Markdown files with readable front matter.
- **Local-first.** KB доступна offline и синхронизируется как user data.
- **Recoverability.** Derived indexes can be rebuilt from primary notes и
  materials.
- **Stability.** Links use stable ids internally, even if title/path changes.
- **Human-readable files.** Filesystem projection must remain understandable
  outside Lumi.
- **Privacy.** Personal KB is private unless user explicitly shares notes or
  generated artifacts.
- **Performance.** Graph and backlinks update incrementally, not by reparsing
  entire vault after every edit.

## Модель данных

```text
KnowledgeBase
  -> KbFolder[]
  -> KbNote[]
  -> KbAttachment[]
  -> DerivedLinkIndex
  -> DerivedGraphIndex
```

Основные сущности:

- `KnowledgeBase` - personal vault-like space.
- `KbFolder` - logical folder.
- `KbNote` - Markdown document.
- `KbNoteRevision` - text revision for sync/merge/history.
- `KbAttachment` - local/resource blob linked from notes.
- `KbLink` - parsed link edge from note body or metadata.
- `KbBacklink` - derived reverse edge.
- `KbGraphNode` - graph node for note/material/tag/concept/artifact.
- `KbGraphEdge` - typed edge.
- `KbSourceRef` - source material/anchor/provenance.
- `KbImportIssue` - unresolved link, missing attachment, parse issue.

Предварительная форма note:

```text
KbNote {
  id
  title
  folder_id
  slug
  body_markdown
  front_matter
  tags
  source_refs
  status: normal | draft | archived | generated_pending_review
  created_at
  updated_at
  revision_id
}
```

Front matter для export/projection:

```yaml
---
lumi_id: kb_...
title: "Idea"
tags: ["reading", "concept"]
source:
  material_id: mat_...
  anchor_id: anch_...
created: "..."
updated: "..."
---
```

## Реализация

### Storage

Primary storage - local Lumi store и sync objects. Файловая система/Obsidian
vault является projection/import channel, not the only database.

Причины:

- web не всегда имеет прямой filesystem access;
- sync/conflicts должны работать одинаково на web, desktop и mobile;
- internal stable ids не должны зависеть от filename;
- graph/search indexes должны строиться из единого хранилища.

### Parsing pipeline

1. Note body changes.
2. Markdown parser extracts front matter, headings, links, tags, embeds.
3. Link resolver maps targets to `KbNote`, `Material`, `Anchor`, `Tag` или
   unresolved link.
4. Derived backlink index updates incrementally.
5. Graph index updates affected nodes/edges.
6. Search index receives note chunks.
7. Sync outbox receives `KbNoteRevision`.

### Graph indexing

Graph index is derived:

```text
KbNote/Material/Annotation/Artifact
  -> parser/entity extractor
  -> KbGraphNode/Edge tables
  -> graph UI/search filters
```

AI-generated edges must carry confidence/source and be visually distinct until
accepted.

### Import from reader

Reader action "добавить в базу знаний":

1. Создает `KbSourceRef` from material + anchor + quote.
2. Создает draft note или вставляет block в existing note.
3. Добавляет source backlink.
4. Индексирует note и source edge.
5. Синхронизирует result.

### Export

Export modes:

- single note Markdown;
- vault folder with notes/assets;
- reader annotations as Markdown notes;
- graph/index JSON sidecar for round-trip.

Export должен оставлять файлы полезными без Lumi: readable front matter,
quotes, links и source metadata.

## Интеграции и зависимости

- **Reader.** Reader creates source refs and annotations that can become KB
  notes.
- **Синхронизация.** KB notes are sync objects with revision/conflict handling.
- **Obsidian.** Obsidian integration reads/writes Markdown projection of KB.
- **Поиск.** KB notes, source refs, tags and graph edges are indexed in unified
  search.
- **Learning.** Flashcards/questions can link to KB notes; mastered concepts
  can update graph metadata.
- **ИИ.** AI creates drafts, summaries, entities and link suggestions.
- **Плагины.** Plugins may add note actions, graph enrichers, import/export
  adapters and custom render blocks through capability grants.

## Альтернативы

- `rejected`: использовать Obsidian vault as primary database. Это плохо
  работает для web/mobile sync, internal ids, conflicts and reader anchors.
- `rejected`: хранить KB как opaque rich-text documents. Markdown нужен для
  переносимости и Obsidian compatibility.
- `rejected`: автоматически принимать все AI-generated notes into KB. Это
  загрязнит базу неподтвержденными связями.
- `revisit`: CRDT editor for KB notes. Полезно для conflict-free editing and
  collaboration, но можно начать с revisions + merge conflicts.
- `revisit`: полноценные typed properties как в Notion/Anytype. Полезно позже,
  но первичный продукт ближе к simple Markdown vault.

- Какой exact syntax использовать для material/anchor links в Markdown export?

  suggestions?
- Нужно ли хранить accepted AI concepts as separate notes или как graph nodes
  без Markdown файла?
